SCHOOL ERP BACKEND - RESTRUCTURING & OPTIMIZATION PLAN
======================================================
This plan outlines the step-by-step process to resolve bottlenecks, fix security flaws, and improve the architectural integrity of the backend.

NOTE: Standardizing file uploads to Cloudinary is currently deferred for a future phase.

PHASE 1: CRITICAL SECURITY & MIDDLEWARE FIXES (Immediate Priority)
------------------------------------------------------------------
[x] 1.1 Remove Token from URL Queries: 
    - Modify `middleware/auth.js`.
    - Remove the `req.query.token` extraction to prevent session hijacking via access logs.
    - Rely strictly on `Authorization: Bearer <token>` and secure HTTP-only cookies.

[x] 1.2 Replace Custom Data Sanitization:
    - Modify `server.js`.
    - Remove the manual regex sanitizer loop.
    - Install and configure `express-mongo-sanitize` and `xss-clean` (or `DOMPurify`) to prevent NoSQL injection and XSS efficiently.

[x] 1.3 Simplify JWT Logic:
    - Modify `utils/jwt.js`.
    - Remove the redundant lazy-loading pattern.
    - Rely on `config/env.js` which already crashes the app if `JWT_SECRET` is missing.

PHASE 2: DATA INTEGRITY & SECURITY UPGRADES
------------------------------------------------------------------
[x] 2.1 Secure Default Password Generation:
    - Modify `admin.student.controller.js`, `admin.teacher.controller.js`, and `admin.registration.controller.js`.
    - Replace predictable passwords (e.g., `Student@1234`) with `crypto.randomBytes` or secure generator.
    - Add a `requiresPasswordChange` boolean flag to the user schemas and enforce it on first login.

[x] 2.2 Implement MongoDB Transactions:
    - Identify controllers handling multi-document writes (e.g., `createStudentWithParent`).
    - Wrap these operations in `mongoose.startSession()` and `session.withTransaction()` to ensure database consistency on failure.

[x] 2.3 Enforce Pagination Limits:
    - Modify `utils/pagination.js`.
    - Hardcode a maximum limit (e.g., `limit = Math.min(parseInt(req.query.limit) || 10, 100)`) to prevent Out-Of-Memory (OOM) crashes from malicious requests.

PHASE 3: PERFORMANCE & NODE.JS BOTTLENECK OPTIMIZATION
------------------------------------------------------------------
[x] 3.1 Implement `.lean()` for Read-Only Queries:
    - Scan all controllers for `find()` and `findOne()` queries that only read data.
    - Append `.lean()` to return plain JavaScript objects, reducing CPU and memory overhead.
    - Apply `select()` to limit returned fields strictly to what the frontend requires.

[x] 3.2 Optimize Search Queries ($regex to $text):
    - Add MongoDB Text Indexes to heavily searched fields (e.g., name, email, studentID) in your Mongoose schemas.
    - Replace inefficient `$regex` queries in controllers with `$text: { $search: searchString }`.

[x] 3.3 Replace Console.logs with Structured Logging:
    - Install `winston` or `pino`.
    - Create a `utils/logger.js` module.
    - Replace all `console.log` statements in controllers and `auth.js` to prevent blocking the Node.js event loop.
    - Set log levels to mute `debug`/`info` in production.

PHASE 4: ARCHITECTURAL ENHANCEMENTS
------------------------------------------------------------------
[x] 4.1 Route-Level Input Validation:
    - Install `zod` or `joi`.
    - Create validation schemas for requests (body, params, query).
    - Apply them as middleware before controllers to clean up business logic (removing manual `if (!name) throw Error` blocks).

[x] 4.2 Introduce Caching Layer:
    - Install `node-cache` (for single instance) or `redis` (for scaling).
    - Cache static or rarely changing queries like Academic Sessions, School Context, and Fee Heads.

[x] 4.3 Decouple Frontend Serving:
    - Modify `server.js` to stop serving static React/Next.js files (`app.use(express.static(...))`).
    - Prepare the frontend to be hosted independently on Vercel, Netlify, or a CDN.

[x] 4.4 Soft Deletes Implementation:
    - Modify schemas to include `isDeleted: { type: Boolean, default: false }` and `deletedAt: Date`.
    - Update delete controllers to toggle this flag rather than using `deleteOne()`.
    - Update core queries to exclude `isDeleted: true`.

PHASE 5: ENTERPRISE FEATURES & ENGINEERING PRACTICES
------------------------------------------------------------------
[x] 5.1 Audit Logging:
    - Create an `AuditLog` schema or Mongoose plugin to automatically track POST, PUT, PATCH, and DELETE actions (Who did what, when).

[x] 5.2 API Documentation:
    - Install `swagger-ui-express` and `swagger-jsdoc`.
    - Document the primary endpoints to assist frontend integration.

[x] 5.3 Automated Testing:
    - Setup `jest` and `supertest`.
    - Write unit/integration tests for critical paths (Auth, Fee Calculation, Multi-Tenant Isolation).

[x] 5.4 CI/CD & Code Quality:
    - Setup `husky` pre-commit hooks for `eslint` and `prettier`.
    - Create a basic GitHub Actions / GitLab CI pipeline to run tests automatically.

PHASE 6: CLOUDINARY INTEGRATION & STANDARDIZATION
------------------------------------------------------------------
[x] 6.1 Setup Cloudinary Config & Helpers:
    - Create `utils/cloudinary.js` to initialize the v2 SDK and export upload/delete helper functions.

[x] 6.2 Standardize Multer Middleware:
    - Create `middleware/upload.js` utilizing `multer-storage-cloudinary`.
    - Configure dynamic folders (e.g., separating profile pictures from documents).

[x] 6.3 Update Controllers & Refactor Uploads:
    - Update controllers (Students, Teachers, Admin, Transport, etc.) to use Cloudinary URLs.
    - Implement deletion logic so old files are purged from Cloudinary when updated/deleted.
